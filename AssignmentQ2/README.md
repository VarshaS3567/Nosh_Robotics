# Q1 – STM32G070RB Switch & LED Blinking

## Objective

Implement a switch-controlled LED blinking application on the STM32G070RB NUCLEO board.

The LED is initially OFF. Each button press changes the LED state according to the following sequence:

| Switch Press | LED State |
|--------------|-----------|
| 1st | Blink at **0.5 Hz** |
| 2nd | Blink at **1 Hz** |
| 3rd | Blink at **2 Hz** |
| 4th | LED OFF |
| 5th | Cycle repeats |

---

## Hardware Used

- STM32 NUCLEO-G070RB
- On-board User LED
- On-board User Push Button
- STM32CubeIDE
- STM32 HAL Library

---

## Software Design

The application uses:

- GPIO External Interrupt (EXTI) for button detection
- Timer interrupt for LED blinking
- State machine for blink frequency control
- Low-power **Sleep Mode** to reduce CPU power consumption

The CPU remains in Sleep Mode whenever no processing is required and wakes only on:

- External button interrupt
- Timer interrupt

---

## Power Optimization

The following techniques are used:

- Sleep Mode (WFI instruction)
- Interrupt-driven operation
- No busy waiting
- CPU wakes only when required

# Q2 – Producer-Consumer Sensor Buffer Simulation

## Objective

Simulate an external sensor that generates random data every second.

Every 10 seconds, the application checks whether at least 50 bytes are available in a shared buffer.

If available:

- Print the latest 50 bytes in hexadecimal
- Remove the printed bytes from the buffer

Otherwise:

- Display that insufficient data is available.

---

## Features

- Random sensor data generation
- Global shared buffer
- Buffer size tracking
- Latest 50-byte extraction
- Hexadecimal output
- Automatic buffer update
- Compatible with online C compilers

---

## Program Flow

1. Every second:
   - Generate 0–5 random bytes.
   - Store bytes in the global buffer.

2. Every 10 seconds:
   - Check buffer size.
   - If buffer contains at least 50 bytes:
     - Print latest 50 bytes.
     - Delete printed bytes.
   - Otherwise:
     - Display "Not enough data".

# Assumptions

## Q1

- The on-board LED is connected correctly.
- The on-board push button generates an external interrupt.
- HAL drivers are available.
- Timer configuration is assumed to be completed using STM32CubeMX.
- Software debounce is not implemented.

---

## Q2

- One simulated second equals one real second.
- Random numbers simulate sensor-generated bytes.
- Maximum buffer capacity is sufficient to avoid overflow.
- Data is stored in a globally accessible array.
- Only the latest 50 bytes are printed.
- After printing, only those printed bytes are removed.
- Program runs for a fixed simulation duration.
